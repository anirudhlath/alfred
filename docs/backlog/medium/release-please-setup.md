# release-please: Versioning, Changelog & QA-Gated Release PRs

**Epic:** [GitHub Chores](../epics/github-chores.md)
**Priority:** medium
**Source:** Branching strategy spec `docs/superpowers/specs/2026-07-18-branching-strategy-design.md` (§6) — supersedes the earlier tag-push release ticket

## Summary
Adopt release-please as the release mechanism: a rolling release PR per repo computes the
semver bump from squash-commit (conventional PR title) history, updates CHANGELOG and
version files, and — when merged — tags `vX.Y.Z` and publishes the GitHub Release.
Merging the release PR is the release act and is manually gated: release milestone empty
→ manual QA pass (`docs/qa-backlog/` tickets for the milestone verified & deleted) →
owner merges. Prod (CachyOS) deploys pinned tags only.

## Context / Motivation
- Hard prerequisite: [branching-conventions-and-pr-title-check](../high/branching-conventions-and-pr-title-check.md)
  — release-please reads conventional squash titles; enforce titles BEFORE enabling it.
- Config per repo (`release-please-config.json` + `.release-please-manifest.json`):
  - alfred: `release-type: python` (updates `pyproject.toml` + CHANGELOG). Note the web
    SPA is not separately versioned — one version per repo.
  - home-service: `release-type: python`.
  - alfred-ios: `release-type: simple` (version file) unless a better Xcode-project
    strategy is verified at implementation.
  - Pre-1.0 semantics: `bump-minor-pre-major: true` (breaking → minor),
    `bump-patch-for-minor-pre-major: false` (feat → minor). Current version seeds from
    the latest existing tag (alfred: v0.1.0).
- **Token:** the workflow must use the bot App token from
  [workflow-chaining-and-bot-commit-resilience](../high/workflow-chaining-and-bot-commit-resilience.md)
  — with `GITHUB_TOKEN`, the tag/Release it creates can never trigger downstream
  workflows (deploy hooks, notifications) added later.
- **Guardrails:** the release PR must be excluded from any auto-merge logic (it is
  merged by the owner only, after QA); its own PR title is generated by release-please
  and must be allowlisted by (or compatible with) the PR-title check; SHA-pin the
  action; `ci-ok` still gates it like any PR.
- The release milestone convention (zero open issues before release) lives in
  [github-issues-agent-work-queue](github-issues-agent-work-queue.md); this ticket only
  consumes it.
- Verify current release-please action inputs/config schema at implementation time
  (context7/docs) — the config surface has evolved across major versions.

## Acceptance Criteria
- [ ] release-please workflow live on alfred: after a `feat:` squash lands, a release PR
  exists proposing a minor bump with correct CHANGELOG and `pyproject.toml` version.
- [ ] Pre-1.0 bump rules verified: a `feat!:` squash proposes **minor**, a `fix:`-only
  history proposes **patch**.
- [ ] Merging the release PR produces the tag + GitHub Release with categorized notes,
  authored via the bot App token.
- [ ] Release PR is never auto-merged (verified against the Dependabot/auto-merge
  workflows) and passes `ci-ok` + the PR-title check.
- [ ] Same setup replicated to home-service and alfred-ios once their CI exists.
- [ ] Release procedure documented in CONTRIBUTING/CLAUDE.md: milestone empty → QA pass
  (qa-backlog cleared for the milestone) → merge release PR → prod deploys the new tag.
